# Issue 10: Settings page tracer

Status: ready-for-agent
Category: enhancement

## Parent

`.scratch/voice-ai-template/PRD.md`

## What to build

A `/settings` page where the user can change their preferred name, choose an OpenAI Realtime voice, and sign out. Wires through the existing preferences layer — no new persistence concepts. Demonstrates the pattern downstream developers will copy for their own settings UIs.

## Acceptance criteria

### `packages/core`

- [ ] Reuse `core.preferences` directly; no new core module needed
- [ ] Define a small `core.preferences.constants` (or similar) listing recognized preference keys (`preferred_name`, `voice`) and their valid values, used both for schema validation and as the source-of-truth for the frontend selector

### `apps/agent`

- [ ] At session start, the worker reads `preferred_name` and `voice` from `core.preferences.list(user)`
- [ ] Voice selection is passed into the realtime model factory
- [ ] If `preferred_name` is set, the system prompt is augmented to address the user by that name

### `apps/api`

- [ ] `GET /preferences` already exists from issue 07; verify it returns `preferred_name` and `voice` when set
- [ ] `PUT /preferences/:key` upserts a single preference (alternative to using the agent tools); validates the value against `core.preferences.constants`

### `apps/web`

- [ ] `/settings` route gated by `RequireAuth`
- [ ] Form (RHF + Zod) with:
  - Preferred name (text input)
  - Voice (shadcn `Select` listing OpenAI Realtime voice options sourced from a shared constants file mirroring the core list)
  - Save button submits via `PUT /preferences/:key` calls (one per changed field)
- [ ] Sign-out button calls `supabase.auth.signOut()` and redirects to `/sign-in`
- [ ] Sidebar nav adds a "Settings" link

### Tests

- [ ] `apps/api` route test for `PUT /preferences/:key` (200 on valid value, 400 on invalid value, 401 without auth)
- [ ] `apps/web` component test for the settings form (validation, sign-out flow)

### Demo

- [ ] User opens `/settings`, sets preferred name to "Sam," picks a different voice, saves
- [ ] On the next conversation, the assistant addresses the user as "Sam" and speaks in the selected voice
