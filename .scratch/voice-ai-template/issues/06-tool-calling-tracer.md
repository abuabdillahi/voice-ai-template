# Issue 06: Tool calling tracer

Status: ready-for-agent
Category: enhancement

## Parent

`.scratch/voice-ai-template/PRD.md`

## What to build

Add the tool-registration and dispatch layer to `core`, then ship two example tools. After this slice, asking the assistant "what time is it?" or "what's the weather in Berlin?" produces correct, real responses via tool calls. The transcript panel surfaces tool invocations so the developer can see which tool was called and what it returned.

This slice establishes the **canonical pattern downstream developers copy** when adding their own domain tools, including the multi-step async HTTP pattern (geocode → forecast), explicit timeouts, and graceful failure handling.

## Acceptance criteria

### `packages/core`

- [ ] `core.tools.tool` decorator registers an async function as a tool, capturing its name, description, JSON-schema arguments (derived from type hints + docstring or pydantic model), and handler
- [ ] `core.tools.registry` exposes `all_tools()` returning the schema list LiveKit Agents expects, and `dispatch(name, args, ctx)` invoking the registered handler
- [ ] Tool handlers receive a `ToolContext` carrying the current `User` plus a logger bound with `tool_name` for structured logging
- [ ] Tool errors are caught at the dispatch layer and returned as `{"error": str}` to the agent rather than raising; the agent's response will reflect the error verbally
- [ ] `core.tools.examples.get_current_time(timezone: str = "UTC") -> str` returns ISO-formatted current time in the requested timezone
- [ ] `core.tools.examples.get_weather(city: str) -> str`:
  - Uses `httpx.AsyncClient` with a 5-second timeout
  - Step 1: calls Open-Meteo geocoding API to resolve city → lat/lon (no API key required)
  - Step 2: calls Open-Meteo forecast API to fetch current temperature and weather code
  - Returns a natural-language summary string
  - Returns a graceful "couldn't find a place called X" string when geocoding finds nothing
  - Catches `httpx.TimeoutException` and `httpx.HTTPError`, returns a readable error string

### `apps/agent`

- [ ] Worker registers all tools from `core.tools.registry` with the LiveKit Agent on session start
- [ ] System prompt updated to mention the available tools and instruct the agent to use them
- [ ] On `function_calls_finished` events, the worker emits a structured log line and forwards a tool-call event into the LiveKit data channel for the frontend
- [ ] Tool calls are passed the authenticated `User` from the room participant identity (set up in issue 05)

### `apps/web`

- [ ] Transcript view distinguishes user / assistant / tool-call entries visually (shadcn `Badge` per role, monospace for tool args/results)
- [ ] Tool-call entries collapse/expand to show args and result (shadcn `Collapsible`)

### Tests

- [ ] `core.tools.examples.get_weather` unit tests using `httpx.MockTransport`:
  - Successful path returns expected summary
  - Geocode returns no results → graceful "couldn't find" message
  - Geocode times out → readable error string
  - Forecast HTTP 500 → readable error string
- [ ] `core.tools.examples.get_current_time` unit tests for known timezone and invalid timezone
- [ ] `core.tools.registry` unit test for register + dispatch round-trip
- [ ] `apps/agent` session test (extending issue 05's harness) scripts a user turn that should trigger `get_weather`; asserts the tool was dispatched and the assistant's spoken response references the result

### Demo

- [ ] User asks "what time is it in Tokyo?" — agent calls `get_current_time` and replies with the correct time
- [ ] User asks "what's the weather in Berlin?" — agent calls `get_weather`, fetches real data from Open-Meteo, and replies
- [ ] User asks "what's the weather in Atlantis?" — agent gracefully reports it couldn't find the place

Deviations: the AC's "harness-based round-trip ('what's the weather in Berlin')" is replaced with a contract test (`test_session_tools.py`) under the brief's own escape hatch — the LiveKit Agents 1.5.x voice-loop test harness is still flagged "evals" upstream and not stable across patch releases. `pytest-asyncio` was added to both `core` and `agent` dev deps (asyncio_mode = "auto") to support the async tool tests without per-test markers.
