Status: ready-for-agent

## Parent

`.scratch/voice-ai-template/PRD.md`

## What to build

Hook LiveKit Agents' `MetricsCollectedEvent` and emit a structured JSON log line per turn capturing server-side time-to-first-audio and component latencies (end-of-utterance delay, LLM time-to-first-token, TTS time-to-first-byte). Set up `structlog` with JSON output, request/conversation correlation IDs, and the per-turn metrics emission. After this slice, `docker logs` on the agent worker shows a queryable stream of latency data per turn.

This is the "minimum viable monitoring" the PRD agreed on. Heavier observability (Langfuse, dashboards, client-side TTFA) is deferred to a future iteration.

## Acceptance criteria

### `packages/core`

- [ ] `core.observability.setup_logging(level: str, json_output: bool)` configures `structlog`:
  - JSON output in production (default), pretty-printed in development
  - Standard processors: timestamp, log level, contextual binding
  - Compatible with stdlib `logging` consumers
- [ ] `core.observability.bind(**fields)` is a thin convenience over `structlog.contextvars.bind_contextvars`
- [ ] `core.observability.handle_metrics_event(event)` accepts a LiveKit `MetricsCollectedEvent` and emits a single structured log line at INFO level with:
  - `event = "turn_metrics"`
  - All numeric fields from the event (eou_delay, llm_ttft, tts_ttfb, total_latency, token counts where available)
  - The currently-bound contextvars (which include `conversation_id`, `session_id`, `user_id`)

### `apps/api`

- [ ] On startup, calls `core.observability.setup_logging`
- [ ] Existing request-id middleware now uses `core.observability.bind(request_id=...)` so subsequent log lines auto-include it

### `apps/agent`

- [ ] On startup, calls `core.observability.setup_logging`
- [ ] Worker subscribes to LiveKit Agents' metrics events and forwards each to `core.observability.handle_metrics_event`
- [ ] On session start (already in issue 09), binds `user_id`, `session_id`, `conversation_id` so those appear on every line for the session
- [ ] On session end, unbinds those vars

### Tests

- [ ] `core.observability.handle_metrics_event` unit test:
  - Given a sample `MetricsCollectedEvent`, asserts a JSON log line is emitted with the expected `event`, numeric fields, and bound context
- [ ] `core.observability` unit test verifying that bound contextvars appear on emitted log lines

### Demo

- [ ] Have a voice conversation
- [ ] `docker logs voice-ai-agent` (or the equivalent compose command) shows a `turn_metrics` JSON line per turn including total_latency in milliseconds
- [ ] The same log lines include `user_id` and `conversation_id`, allowing a user to grep the log for a single conversation
- [ ] `docs/agents` does **not** require updates; the README's "Observability" section documents how to grep for `turn_metrics` and notes that a future iteration will add Langfuse and a dashboard
