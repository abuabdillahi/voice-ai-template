## What to build

The first end-to-end vertical tracer: Supabase authentication wired through every layer. After this slice, a developer cloning the template, signing up via the web UI, and landing on the home page sees their email address rendered — proving Supabase Auth, the typed FastAPI dependency, the Vite frontend, generated TS types, and Supabase Cloud's free tier all work together.

This slice establishes the patterns every subsequent tracer reuses: Supabase migration, RLS policy template, `core` module shape, FastAPI route handler shape, generated OpenAPI client shape, TanStack Router route shape, test patterns at every layer.

## Acceptance criteria

### Schema and Supabase setup

- [ ] `supabase/` directory initialized via Supabase CLI; `supabase/config.toml` committed
- [ ] `supabase/migrations/` contains an initial migration that enables the `pgvector` extension and creates a placeholder for future tables (no application tables here yet — auth tables are Supabase-managed)
- [ ] `supabase start` brings the local Supabase stack up successfully

### `packages/core`

- [ ] `core.config.Settings` (pydantic settings) loads `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_JWT_SECRET` from environment
- [ ] `core.auth.verify_token(token: str) -> User` validates Supabase JWTs (JWKS or HS256 depending on Supabase config) and returns a typed `User` dataclass with `id` (uuid) and `email`
- [ ] `core.auth.get_current_user` is exported as a FastAPI `Depends`-compatible callable that extracts the bearer token, verifies it, and returns `User`; raises 401 on failure with structured detail

### `apps/api`

- [ ] FastAPI application factory with CORS configured for the dev frontend origin
- [ ] `GET /health` (existing from issue 03) and `GET /me` route returning the current `User` (200) or 401 if unauthenticated
- [ ] OpenAPI schema available at `/openapi.json`
- [ ] Request-ID middleware binds a `request_id` to the structlog context for each request

### `apps/web`

- [ ] Supabase JS client initialized with public env vars (`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`)
- [ ] TanStack Router file-based routes: `/sign-in` (custom shadcn form using `signInWithPassword` and `signUp`) and `/` (home, requires auth, fetches `/me`, displays the email)
- [ ] An `AuthProvider` exposes the current Supabase session via context and a `useUser` hook
- [ ] A `RequireAuth` route guard redirects unauthenticated users to `/sign-in`
- [ ] A typed API client generated from `apps/api/openapi.json` using `openapi-typescript`, output to `apps/web/src/api/types.gen.ts`; a small `apiFetch` helper attaches the Supabase access token as a bearer header
- [ ] `pnpm gen:api` script regenerates the types against a running api
- [ ] `apps/web` integrates Tailwind, shadcn (initialized with components.json), and at least the shadcn `Button`, `Input`, `Card`, `Form` components used by `/sign-in`

### Tests

- [ ] `core.auth` unit tests: valid token → User, expired token → raises, malformed token → raises, missing claims → raises
- [ ] `apps/api` unit test for `/me`: 200 with valid mocked user dependency, 401 without
- [ ] `apps/web` example component test using vitest + React Testing Library on the sign-in form
- [ ] All run via `turbo run test`

### Demo

- [ ] On a fresh clone with Supabase env vars configured, running the dev stack and registering a new user lets the user sign in and see their email on the home page
- [ ] README updated with the auth setup section (Supabase project creation steps, env vars to set)
